## 1. FPGA 基础概念与启动模式

### 启动模式
*   **JTAG**：对应下载程序模式（最常用的在线调试与烧录模式）。
*   **NAND**：NAND Flash 启动（部分板卡不支持）。
*   **OSPI**：出厂设置启动模式（高速 Flash 启动）。
*   **SDcard**：TF 卡启动模式。

### 语言本质区别
*   **Verilog** 是**并行语言**，用于设计生成物理电路。
*   **C 语言** 是**串行语言**，用于生成由 CPU 顺序执行的机器指令。

> 💡 **硬件时钟常识**：
> FPGA 板子晶振若是 `50MHz`，意味着 FPGA 的“心脏”每秒钟跳动 50,000,000 次。换算成周期就是 **每 20纳秒 (ns)**，代码里的系统时钟计数器就会加 1。
> 实际物理时间的计算公式：`T = 计数值 * 20ns`。
> *(注：Vivado 内部还支持温度检测等硬件级监控功能。)*

## 2. Vivado 开发基础流程

1.  **创建工程 (Create Project)**：创建工程文件夹并添加源文件。根据查表找到对应的芯片型号进行设置和绑定（例如我的板子对应的是 `xc7020-2啥的`）。
2.  **编写代码**：设置好后进入编辑环境，写好对应程序代码，`Ctrl+S` 保存，软件会自动判别是否有语法错误。
3.  **引脚分配与约束 (Open Elaborated Design)**：
    *   点击 `Run Analysis` -> `Open Elaborated Design`，软件进入模拟程序环境。如果没有约束文件（将程序变量绑定在对应引脚的文件），需创建并编写。有两种方式：
        *   *第一种*：通过查表（在`文件夹A-3`中的底板原理图和核心板原理图中）查询芯片引脚表和对应的电压参数，填写到约束文件中。
        *   *第二种*：打开资料盘的 `.xdc` 文件，拷贝约束代码，进行特定化引脚编辑后在约束文件中粘贴。
    *   *(注：关于为什么这几个测试时钟要绑定在 `j3` 模块显示走线，需结合具体硬件电路图分析。)*
4.  **生成比特流 (Generate Bitstream)**：软件会根据固定流程自动执行综合和实现，最后生成比特流文件。
5.  **烧录验证 (Program Device)**：将板子设定为 **`on/on` 状态**，接入数据下载器。通电后点击 `Open Hardware Manager` 自动连接，上部蓝色词条提示后下载对应文件到板子中，完成功能实现。

## 3. Verilog 语法基础与规则

### 进制表示与位宽
*   格式：`[位宽]'[进制][数值]`。中括号给出的是位宽，高位在前低位在后。
*   `b`：二进制 | `o`：八进制 | `d`：十进制 | `h`：十六进制。
*   *示例*：`4'b0101` 表示四位二进制数字 0101；`16'h9AA9` 表示16位十六进制数字。
*   *默认规则*：若未给出位宽，默认位宽为 32 位（未定义位宽变量则默认为 1）；若进制也没有，默认 32 位十进制 (`d`)。

### 变量类型与赋值核心口诀
> 📌 **一句话口诀：**
> 用 `assign` 赋值的（连续赋值），必须用 **`wire`**。
> 在 `always` 或 `initial` 里面赋值的，必须声明为 **`reg`**。

*   `reg` 类型：只有在 `always` 和 `initial` 语句中可以被赋值。
*   `wire` 类型：不能在 always 中被赋值，其值由驱动它的元件决定。若没有驱动元件连接，默认值为高阻态 (`Z`)。
*   在 `input` 和 `output` 中定义的 `reg` 和 `wire` 意思是这个端口在模块中可以以相应的类型进行赋值（默认缺省是 `wire`）。

### 运算符与基础语法
*   **基础运算**：`+`, `-`, `*`, `/`, `>`, `<`, `>=`, `<=`, `!=` (与 C 语言基本相同)。
*   **逻辑运算**：`!` (非), `&&` (与), `||` (或)。
*   **条件运算符**：`判断条件 ? a : b` (条件为真选 a，条件为假选 b)。
*   **位运算补零**：对应位进行运算，若低位宽与高位宽进行位运算，会给低位宽的数据进行补 0。
*   **模块传参**：`.clk (sys_clk)`，就是将顶层的 `sys_clk` 信号与底层的 `clk` 信号进行传递，模块的值也可以通过这个语法进行传递。
*   **注释**：Verilog 注释方式与 C 语言相同（`//` 或 `/* */`）。
*   `localparam`：本地定义一个状态/常量（类似于 C 语言的 `#define`），常用在 `case` 语句中。
*   `#` 符号：延时符号（多用于 Testbench 仿真）。

### 阻塞与非阻塞赋值、触发模式
*   `posedge` (上升沿)，`negedge` (下降沿)。沿触发常常用来描述时序逻辑。
*   没有 `posedge` 则是电平触发（直接用 `*` 表示后面语句中所有的输入变量都是敏感的）。
*   **阻塞赋值 (`=`)**：后一句赋值语句是在前一句赋值语句结束后才开始赋值。
*   **非阻塞赋值 (`<=`)**：不同语句的赋值是可以同时进行的。*(注意：有没有 `<=` 对于是否是阻塞赋值很关键，`<=` 在这里并不是比较符号而是赋值符号！)*
*   **可综合模块**：可以生成物理电路。**不可综合模块**：用于仿真，不可生成物理电路。

### 语法补充：位拼接运算符 `{}`
拼接运算符就是位数相加并赋值给新的变量。它的作用就像胶水一样，把几个小的信号或数据“首尾相连”拼成一个更宽的信号。
*语法详解：`{3'b000, ram_addr}`*
*   `{ }`：拼接的大括号，表示里面的东西要连在一起。
*   `,`：分隔符，隔开每一部分。
*   `3'b000`：高位；`ram_addr`：低位（假设是 5 位宽）。

```text
  3'b000        ram_addr (5 bits)
  +-------+     +-----------------+
  | 0 0 0 |     | 0   0   1   0   1 |
  +-------+     +-----------------+
      |                  |
      +--------+---------+
               |
               v
         0 0 0 0 0 1 0 1  (拼成了一个8位宽的数据，即 8'd5)
```

## 4. 核心逻辑设计代码实例

### 4.1 0.5s 脉冲定时器
整个流程就像跑步：
1. **起跑**：复位后，`timer_cnt` 是 0。
2. **跑步中**：每个时钟上升沿（每 20ns），没数满就走 `else` 分支加 1（`0 -> 1 ... -> 24,999,998`）。
3. **撞线与下一轮**：当变成 `24,999,999` 时条件成立，下一个时钟到来时强制清零，同时可触发状态机的跳转。*(比如与其他代码组合，用于流水灯代码，检测到满值就给当前状态加 1，不满足就继续增加计时器。)*

```verilog
reg [25:0] timer_cnt; // 定义一个足够大的寄存器来装这个数
always @(posedge sys_clk or negedge sys_rst_n) begin
    if (!sys_rst_n)                 // 1. 如果复位键按下，计数器归零
        timer_cnt <= 26'd0;         
    
    // 2. 核心判断：数满了吗？ (注意这里是 CNT_MAX - 1，也就是 24,999,999)
    else if (timer_cnt == CNT_MAX - 1) begin 
        timer_cnt <= 26'd0;         // A. 数满了：立刻归零，准备下一轮
        // (在这个时刻，state_cnt 会在另一个 always 块里加 1)
    end
    else                            // 3. 没数满
        timer_cnt <= timer_cnt + 1'b1; // B. 继续数数：+1
end
```

### 4.2 按键消抖与“打拍”原理（消除亚稳态与边沿检测）
在 FPGA 设计中，“打拍”（Registering / Pipelining）是核心基本功。简单来说，“打拍”就是把信号存入寄存器（D触发器）里，等下一个时钟边沿到来时再输出。打一拍就是延迟一个时钟周期，打两拍就是延迟两个。针对按键这种应用场景，打拍主要有两个目的：

**一、“打两拍”：为了安全（消除亚稳态）**
原理：按键是异步信号，按下的瞬间可能卡在时钟上升沿的“建立/保持时间”窗口内，D触发器就会不知所措，输出一个既不是高电平也不是低电平的怪异状态，导致在 0 和 1 之间震荡，产生“亚稳态”。如果这个烂信号直接参与逻辑运算，整个电路就会出错。
*(比喻：就像接一个空中飞来的球。如果你直接用手去抓第一拍，可能会手滑抓不稳；但如果你先把球接到手套里缓冲一下，再拿给队友作为第二拍，球就稳了。)*

```verilog
reg key_r0, key_r1;
always @(posedge sys_clk) begin
    key_r0 <= key_in;   // 第一拍：捕捉外部信号，可能包含亚稳态
    key_r1 <= key_r0;   // 第二拍：亚稳态基本消除，变成干净安全的 0 或 1
end
```

**二、“打一拍”+“前一拍”：为了抓边沿（边沿检测）**
原理：FPGA 是数字电路，它只知道当前时刻是 0 还是 1。它不知道“刚才”是什么。为了检测按键按下或松开跳变的瞬间，需要保留“今生”和“前世”的状态。

```verilog
reg key_d0; // 现在的状态 (Current - 今生)
reg key_d1; // 上一次的状态 (Previous/Old - 前世)
always @(posedge sys_clk or negedge sys_rst_n) begin
    if(!sys_rst_n) begin
        key_d0 <= 1'b1;
        key_d1 <= 1'b1;
    end
    else begin
        key_d0 <= key_in;  // 第一拍：记录最新值
        key_d1 <= key_d0;  // 第二拍：记录老值，把上一刻的 d0 存起来 (打一拍加前一拍)
    end
end
// 组合逻辑判断：前世是1，今生是0 -> 触发下降沿 (实体按键按下)
assign key_negedge = (key_d1 == 1'b1) && (key_d0 == 1'b0); 
```

> 📝 **消抖与按键硬件常识**：
> *   **消抖逻辑**：大致就是按下按钮之后给一个计时器赋值，然后等 20ms 后如果按键没出现变化，就会给最终的 key 赋值真值。我的写法：新增一个检测变量，当按键被按下时 key in 变为 1，不等于检测变量，开始 20ms 的计时。如果计时跑满则将输入量传递给检测变量（最终达到相同的值的判定条件，计时器归 0）。如果在此期间发生抖动，在极小时间内再次触发了，就会重新满足判别条件并重新开始计时，按键真值并没有被传递下去。
> *   **按键电平**：实体开关是按下低，松开高。触摸按键通过 IC 芯片检测电容容值变化，我所使用的板子为 **`00模式`**，即与实体按键一样按下低，松开高。编写代码时反转一下语句就可以做到了。对于按下/按住都可以这样写，关键在于后续传递的最后检测变量怎么用。
> *   **蜂鸣器**：为 1 时鸣叫，为 0 时不叫。

### 4.3 PWM 呼吸灯设计
**核心原理**：如果要进行 PWM 调光，我们需要先让系统生成一个 PWM 波（实际上 PWM 就是通过灯光的亮对于整个 PWM 时间的占空比多少来实现的亮度变化）。`pwm-1` 就是整个 PWM 波的最大位，变量 `pwm_cnt` 不断跟随系统时钟在曲线上从 0 到最大值循环。

此时，如果我们给这条曲线**突然插入一条 `y = state`（就像一把刀插进 PWM 波里面）**，将曲线截成两段：让 `pwm_cnt < state` 的部分灯亮，大于的部分暗，这样就能实现占空比调光。当然，这是一个小周期上的极速亮暗。如果我们把时间尺度放大到整个呼吸周期来看，对这些亮暗进行积分，那么效果就可以做到宏观上的变亮/变暗了。

```verilog
// --- A. 生成基础 PWM 波 ---
if(pwm_cnt >= pwm_1 - 1)
    pwm_cnt <= 16'd0;
else
    pwm_cnt <= pwm_cnt + 1'b1;

// --- B. 呼吸节拍控制 (这里就是整个对时间积分的大周期累加语句) ---
if(step >= breathe_speed - 1) begin 
    step <= 16'd0; 
    if(now == 1'b0) begin 
        // [模式：变亮]
        if(state >= pwm_1) 
            now <= 1'b1; 
        else
            state <= state + 1'b1; 
    end
    else begin 
        // [模式：变暗]
        if(state == 16'd0) begin 
            now <= 1'b0;  
            now_led <= ~now_led; 
        end
        else begin
            state <= state - 1'b1; 
        end
    end
end
else begin
    step <= step + 1'b1; 
end

// --- C. 最终判别：一把刀插进PWM波里面，实现占空比调光 ---
assign led_out = (now_led == 1'b0) ? ((pwm_cnt < state) ? 1'b1 : 1'b0) : 1'b0;
```

## 5. 在线调试 (ILA) 与 Vivado 进阶

*   **Mark Debug 属性**：网表文件在线调试时，在需要调试的变量名前面添加代码：`(* mark_debug = "true" *) reg 变量名;`。注意是在定义名前面添加，添加 Debug 属性之后会自动进这里，还可以**防止 Vivado 对这个软件/变量进行优化**。
*   **Set up Debug (设置探针)**：在 Synthesis（综合）里添加 ILA IP 核，将对应信号进行抓取。
    *   `Numbers of probes`：探针数量，往往代表除去系统时钟以外的其他 reg/wire/input/output 型变量。
    *   `Sample data depth`：数据样本深度（波形抓取长度）。
*   **触发调试**：设置好生成比特流后就可以进入在线 Debug。打开调试模块后，将对应变量拖入 `Trigger`，可以进行对应的触发模式设置。
*   **编译模式**：
    *   `OOC (Out of Context)`：IP 核作为独立模块编译。生成比特流全局编译时可省略对该 ILA/IP 的编译，节约时间。
    *   `Global`：全编译模式。

## 6. IP 核深度解析 (PLL / BRAM / FIFO)

IP 核就是一个预先设置好的功能模块（当然也可以自定义模块功能）。厂商给出的 IP 核通常无法直接修改代码，当我们要使用时，点击 `IP Sources` 选择 `.veo` 文件，查看帮我们例化好的模板并绑定对应的输入输出端口即可。这大大简化了复杂且常用功能的编写难度。

> ⚠️ **重要避坑**：IP 核复位时钟通常是**高电平触发**，而系统复位信号是低电平触发，所以在传入 `reset` 信号里面要取反（如 `~sys_rst_n`）。

### 6.1 Clocking Wizard (时钟向导 - PLL/MMCM)
*   可以根据时钟输入频率（板载频率）设置输出时钟，要多少设置多少。
*   **PLL**：对时钟进行管理（输出各种频率的时钟）。
*   **MMCM**：混合模式的时钟管理（内部包含了 PLL 和一个用于控制相位的电路）。

### 6.2 BRAM (块 RAM - 随机访问存储器)
写入逻辑：将数据通过 `DINA` 写入到对应地址 `ADDRA` 中。读出逻辑：通过读地址 `ADDRA` 中的数据经过 `DOUTA` 读出。
*   `DINA`：数据写入端口。
*   `ADDRA`：地址端口。
*   `DOUTA`：读出数据端口。
*   `WEA`：读写控制信号（`1` 写入，`0` 读出）。**注意：IP核要求 wea 是向量，所以即使 1 位也要写成 `wire [0:0] ram_we;`**
*   `ENA`：使能端口（为高电平才能进行读写）。
*   `REGCEA`：寄存器使能端口（为高电平才能进行读写改变，低电平保持不变）。
*   `RSTA`：复位端口。
*   **双端口 RAM 注意事项**：双端口其实跟单端口差不多，但**绝对不能同时对一个地址进行读写操作**，否则会出现冲突。必须要有一个 `rd_flag` 信号来控制端口的读写。
*   **无输出报错解决**：因为如果本次设计的 RAM 没有输出端口，直接生成比特流会报错。我们需要在 Synthesis 里面添加 IP 核（Set up Debug）将对应的信号进行抓取，然后才能正确地生成比特流。

### 6.3 FIFO (先进先出缓存)
FIFO 是对 RAM 的一层封装，按顺序写入和读出，**没有地址线**。
**核心应用：跨时钟域信号传递（异步 FIFO）**。例如将摄像头的时钟写入 FIFO，然后内存通过读 FIFO，两个区块的时钟分别和两边同步，从而做到跨时钟传递。

**重要端口与信号标志位：**
*   **写入控制**：给写使能，开始写入数据。
*   `full`：满信号，所有缓存器都缓存了数据，满信号传回说明满了，不能再写入了。
*   `almost full`：接近于满，再写一个就满了。
*   `prog full`：可编程阈值满信号。例如最大值为 512，我设置为 256，到了 256 就拉高。（`prog empty` 可编程空信号上下阈值同理）。
*   `injectxxxx` 端口：ECC 模式，可以进行错误写入（人为注入）测试。
*   **读出控制**：
*   `empty`：读出空信号，同理满信号。
*   `valid`：读出有效信号。
*   `underflow`：空读出信号（在空状态下强制读出的报错）。
*   `rd_data_count` / `wr_data_count`：当前读/写缓存数量。
*   **预读模式 (FWFT)**：异步 FIFO 读出标准有一拍读延迟。如果等不了这一拍的延迟，可以用**预读模式**（`d0` 预读），读延迟提前一拍。

## 9. 通信协议基础知识 (FPGA/单片机核心)

在 FPGA 或单片机开发中，芯片与外设（如传感器、屏幕）或芯片与芯片之间的数据交互，都离不开通信协议。根据这几张图的要点，通信基础知识可以归纳为以下四个维度：

### 9.1 串行通信 vs 并行通信 (按数据通信方式分类)
*   **串行通信 (Serial)**
    *   **原理**：数据像排队一样，按位（Bit）顺序，一位一位地在一条数据线上进行传输（如 `0 1 0 1 1 0 1 0` 依次发送）。
    *   **优点**：占用引脚（I/O）少，布线简单，成本低。
    *   **缺点**：传输速率相对较低。
*   **并行通信 (Parallel)**
    *   **原理**：数据的各个位（如 8 位数据）在多条数据线上同时进行传输（大家并排一起走）。
    *   **优点**：传输速率高。
    *   **缺点**：占用引脚（I/O）资源多，布线复杂，长距离传输容易产生干扰，成本高。

### 9.2 单工 / 半双工 / 全双工 (按数据传输方向分类)
*   **单工通信 (Simplex)**：数据只能沿**一个方向**传输。
    *   *(比喻：就像“收音机广播”，电台只能发，你只能收。)*
*   **半双工通信 (Half-Duplex)**：数据可以沿**两个方向**传输，但**需要分时进行**（不能同时）。
    *   *(比喻：就像“对讲机”，你说的时候我听，我说的时候你听。)*
*   **全双工通信 (Full-Duplex)**：数据可以**同时进行双向传输**。
    *   *(比喻：就像“打电话”，双方可以同时说话和倾听，互不干扰。)*

### 9.3 同步通信 vs 异步通信 (按数据同步方式分类)
*   **同步通信 (Synchronous)**
    *   **核心特征**：**带有单独的时钟信号线**。
    *   **原理**：发送方和接收方共用一根时钟线（Clock），接收设备在时钟信号的跳变沿（如上升沿或下降沿）准确地读取数据线上的电平状态。
*   **异步通信 (Asynchronous)**
    *   **核心特征**：**没有时钟端口**，发送方和接收方使用各自独立的内部时钟。
    *   **原理**：因为没有时钟线来对齐节奏，所以必须把主体数据进行打包，以“**数据帧**”的格式传输。直接在数据信号中穿插一些同步用的信号位。
    *   **👉 UART 串口通信示例**：
        1. 必须规定 **起始位**、**数据位**（通常按 LSB 最低位 到 MSB 最高位顺序）、**奇偶校验位**、**停止位** 等。
        2. 通信双方**必须提前约定好数据的传输速率（即波特率 Baud Rate）**，这样接收方才能按预定的节奏去“抓取”数据。

### 9.4 常见串行通信接口大汇总
在实际工程开发中，常用的串行通信协议主要有以下四种，遇到对应的引脚名就能大概猜到它用的什么协议：

| 通信标准 | 引脚说明 | 通信方式 | 通信方向 |
| :--- | :--- | :--- | :--- |
| **UART** <br>*(通用异步收发器)* | **TXD**: 发送端 <br>**RXD**: 接收端 <br>**GND**: 公共地 | **异步通信** | **全双工** |
| **1-WIRE** <br>*(单总线)* | **DQ**: 发送 / 接收端 | **异步通信** | **半双工** |
| **IIC (I2C)** <br>*(内部集成电路)* | **SCL**: 同步时钟 <br>**SDA**: 数据输入 / 输出端 | **同步通信** | **半双工** |
| **SPI** <br>*(串行外设接口)* | **SCK**: 同步时钟 <br>**MISO**: 主机输入，从机输出 <br>**MOSI**: 主机输出，从机输入 <br>**CS**: 片选信号 | **同步通信** | **全双工** |

> 💡 **学习与实战建议**：
> 1. **UART** 是开发板上最常见的调试接口，通常用一条 USB 线连电脑就能看串口打印信息。
> 2. **IIC** 常常用来连一些对速度要求不高的传感器（比如温湿度传感器、EEPROM）。
> 3. **SPI** 速度快，一般用来连接需要刷数据的设备（比如 TFT 液晶屏、Flash 存储芯片）。
## 10. UART 串口通信深度解析 (硬件层与协议层)

UART (Universal Asynchronous Receiver-Transmitter，通用异步收发器) 也就是我们常说的“串口”。它的核心工作原理是：**发送时将并行数据转换成串行数据发出去，接收时将串行数据转换回并行数据。**

学习 UART 必须从两个层面来理解：**物理层（线怎么连、电平是多少）** 和 **协议层（数据帧长什么样、速度有多快）**。

### 10.1 物理层：接口、连线与电平转换

#### 1. 核心引脚与连线规则
虽然 RS232（DB9 接口）有 9 个引脚，但在绝大多数单片机和 FPGA 开发中，我们只需要用到最核心的 **3 根线**：
*   **TXD** (Transmit Data)：发送数据端。
*   **RXD** (Receive Data)：接收数据端。
*   **GND** (Ground)：公共地线。

> 💡 **连线铁律（交叉连接）**：
> 设备 A 和设备 B 通信时，必须是：**A 的 TXD 接 B 的 RXD；A 的 RXD 接 B 的 TXD；GND 必须接 GND（共地是通信的前提！）**。

#### 2. 致命大坑：电平标准不匹配
这是新手最容易烧板子或通信失败的地方！
*   **FPGA / 单片机的电平 (TTL/CMOS)**：逻辑 1 = 3.3V/5V，逻辑 0 = 0V。
*   **传统电脑串口的电平 (RS-232)**：逻辑 1 = -15V \~ -3V，逻辑 0 = +3V \~ +15V。
*   **结论**：**TTL/CMOS 绝对不能和 RS-232 接口直接相连！** 否则不仅无法识别，甚至可能烧毁芯片。

*(比喻：FPGA 讲中文，传统串口讲英文，两人直接对话完全听不懂，必须找个“翻译官”。)*

#### 3. 如何解决电平与接口问题？
由于现在的电脑基本没有 DB9 接口了，通常有以下两种“翻译”方案：
*   **方案一（板载转换芯片）**：开发板上自带了 `CH340` 等 USB 转串口芯片。我们只需要用一根普通的 **Type-C / USB 线** 连接电脑和开发板即可。（电脑端需要安装 CH340 驱动）。
*   **方案二（外接转换模块）**：开发板只有纯粹的 TTL 排针，电脑端使用一根带有转码芯片的 **USB 转 TTL 线** 进行连接。

---

### 10.2 协议层：数据帧格式 (Data Frame)

UART 是异步通信，没有时钟线，所以发送数据必须严格按照“包”的格式来，这个包就叫**数据帧**。平时线上保持**高电平（空闲状态）**。

一帧完整的数据包含以下几个部分（按发送顺序）：
1.  **起始位 (Start Bit)**：必须是 **1 个比特位的低电平 (0)**。它的作用是告诉接收方：“我要开始发数据了，准备接收！”。
2.  **数据位 (Data Bits)**：传输的有效数据。通常是 **8 位**（1个字节）。
    *   **传输顺序是 LSB 在前，MSB 在后**（即先发最低位，最后发最高位）。例如发送 `0x5A` (二进制 `01011010`)，线路上的实际电平顺序是从右往左发的：`0 -> 1 -> 0 -> 1 -> 1 -> 0 -> 1 -> 0`。
3.  **奇偶校验位 (Parity Bit)**：可选。占用 1 个比特位，用于检测数据在传输过程中是否出错（一般配置为“无校验”）。
4.  **停止位 (Stop Bit)**：一帧的结束标志。必须是 **逻辑高电平 (1)**。可以是 0.5、1、1.5 或 2 个比特位（通常配置为 1 位）。

> 📝 **口诀**：空闲为高，拉低就起跑；发完数据（LSB起头），拉高就停表。

---

### 10.3 传输速率：波特率 (Baud Rate)

既然没有时钟线来对齐节奏，收发双方怎么知道一个信号要持续多久呢？这就需要约定**波特率**。

*   **定义**：串口通信的速率，表示**每秒钟传输二进制数据的位数**，单位是 **bps (bit per second, 位/秒)**。
*   **常用波特率**：`9600`、`19200`、`38400`、`57600`、**`115200`** (目前最常用)。
*   **关键要求**：**通信双方的波特率必须完全一致！** 如果 FPGA 设置的是 115200，而电脑串口助手设的是 9600，接收到的绝对是一堆乱码。

*(换算常识：如果波特率是 115200 bps，那么传输 1 个 Bit 所需要的时间就是 `1 / 115200` 秒，大约为 8.68 微秒。这就是写 FPGA 串口代码时，计数器需要计数的时长标准！)*
## 10.4 UART 串口收发模块的 RTL 设计与架构 (RX/TX/TOP)

在前面已经理解了 UART 的**物理层**（TX、RX、GND、电平标准）和**协议层**（起始位、数据位、停止位、波特率）之后，接下来就进入最关键的工程实操：**如何在 FPGA 里真正把 UART 的接收模块、发送模块和顶层模块写出来**。

这一部分不仅记录“代码怎么写”，更重要的是理解：
1. **为什么要这样分模块；**
2. **RX、TX、TOP 各自负责什么；**
3. **修改前哪里容易写错；**
4. **纠错后的代码为什么是对的。**

---

### 10.4.1 UART 工程的整体结构

一个最基础、最经典的 UART 工程，通常分成 3 个核心模块：

- **`uart_rx`**：串口接收模块  
  **作用**：把外部 `uart_rx` 引脚上进来的**串行数据**，按照波特率节拍一位一位采样，最后拼成 1 个字节的**并行数据**。
- **`uart_tx`**：串口发送模块  
  **作用**：把输入的 1 个字节**并行数据**，加上起始位和停止位，再按照波特率节拍一位一位从 `uart_tx` 引脚送出去。
- **`uart_top`**：顶层模块  
  **作用**：作为整个工程的“总接线员”，负责：
  1. 对外暴露 FPGA 真正的物理引脚接口；
  2. 例化 `uart_rx` 和 `uart_tx`；
  3. 用内部 `wire` 把两个子模块连起来。

> 💡 **一句话理解：** `uart_rx` 负责“听”，`uart_tx` 负责“说”，`uart_top` 负责“搭线”。

---

### 10.4.2 实操目标：串口回环 (Loopback)

本节最核心的实操功能是：**串口回环测试**。

**工作流程**：
1. 电脑通过串口助手发 1 个字节给 FPGA；
2. FPGA 的 `uart_rx` 把这个字节接收下来；
3. 接收完成后，立刻把这个字节交给 `uart_tx`；
4. `uart_tx` 再把同样的数据原样发回给电脑。

如果电脑发出去的是 `A`，收到的也是 `A`，说明 RX 和 TX 两端通信完全正常。
这个实验一次性验证了：**波特率是否准确、起始位/停止位逻辑是否正确、顶层连接是否正确、XDC 物理约束是否匹配**。

---

### 10.4.3 修改前的常见错误写法（Top层极易踩坑）

在实际初学过程中，最常见的问题不是 RX、TX 代码逻辑本身，而是**Top层命名混乱、与物理约束不一致**。下面是一个“容易出错”的旧版示例：

```verilog
// ❌ 修改前：命名随意，极易与 XDC 约束文件不匹配
module top(
    input  wire clk,
    input  wire rst_n,
    input  wire rx,
    output wire tx
);

    wire [7:0] data;
    wire       flag;

    uart_rx u_uart_rx(
        .sys_clk   (clk),
        .sys_rst_n (rst_n),
        .uart_rx   (rx),
        .rx_data   (data),
        .rx_done   (flag)
    );

    uart_tx u_uart_tx(
        .sys_clk   (clk),
        .sys_rst_n (rst_n),
        .tx_data   (data),
        .tx_en     (flag),
        .uart_tx   (tx)
    );

endmodule
```
#### 为什么这是错误的/不规范的？

- **顶层端口命名和 XDC 不一致**：如果 XDC 中绑定的引脚名是 `sys_clk` 和 `uart_rx`，但 Top 层写的是 `clk` 和 `rx`，Vivado 在综合或实现时会报错（如 `Unconstrained Logical Port`）。
- **模块名不规范**：顶层直接叫 `top`，在包含多个实验的综合工程中极易引发模块名冲突。
- **连线命名缺乏语义**：内部 `wire flag;` 语义不明，日后维护难以一眼看出这是“接收完成标志位”。

---

### 10.4.4 修改后的规范 Top 层代码

下面是纠错后的规范写法，严格对齐物理约束与语义规范：

```verilog
// ✅ 修改后：命名规范，端口与 XDC 一一对应，语义清晰
module uart_top(
    input  wire sys_clk,    // 系统时钟，匹配 XDC
    input  wire sys_rst_n,  // 系统复位，低电平有效，匹配 XDC
    input  wire uart_rx,    // UART 接收引脚，匹配 XDC
    output wire uart_tx     // UART 发送引脚，匹配 XDC
);

    // 内部连接线：语义明确
    wire [7:0] rx_data;     // 接收到的 8-bit 数据
    wire       rx_done;     // 接收完成脉冲标志

    // -----------------------------------------------------------------
    // 例化 UART 接收模块
    // -----------------------------------------------------------------
    uart_rx u_uart_rx(
        .sys_clk   (sys_clk),
        .sys_rst_n (sys_rst_n),
        .uart_rx   (uart_rx),
        
        .rx_data   (rx_data), // 输出：接收到的数据
        .rx_done   (rx_done)  // 输出：接收完成标志
    );

    // -----------------------------------------------------------------
    // 例化 UART 发送模块
    // -----------------------------------------------------------------
    uart_tx u_uart_tx(
        .sys_clk   (sys_clk),
        .sys_rst_n (sys_rst_n),
        .tx_data   (rx_data), // 输入：将接收到的数据直接给发送端
        .tx_en     (rx_done), // 输入：接收完成脉冲直接作为发送使能脉冲
        
        .uart_tx   (uart_tx)  // 输出：发送到物理引脚
    );

endmodule
```
### 10.4.5 前后差异总结：我这次到底改对了什么？

这次修改不仅是“改了个名字”，而是把工程结构、代码可读性和物理接口规范一起修正了。

| 对比项 | 修改前的旧代码 | 修改后的规范代码 | 纠错本质与优势 |
| :--- | :--- | :--- | :--- |
| **顶层模块名** | `top` | `uart_top` | 语义明确，便于多模块复杂工程的管理 |
| **时钟/复位端口** | `clk` / `rst_n` | `sys_clk` / `sys_rst_n` | **与 XDC 强一致**，避免管脚未约束报错 |
| **收发端口名** | `rx` / `tx` | `uart_rx` / `uart_tx` | **与 XDC 强一致**，直接映射物理硬件 |
| **内部连线名** | `data` / `flag` | `rx_data` / `rx_done` | 见名知意，提高 RTL 代码的可维护性 |
| **工程可靠性** | 极低（易报错） | 高（标准企业级写法） | 逻辑正确 + 接口正确 = 成功上板验证 |

### 10.4.6 纠错后的 Top 层代码原理解析（回环的核心）

仔细观察 `uart_top` 的例化逻辑，整个串口回环的核心机制仅靠两根线实现：

1. **`rx_data` 接 `tx_data`**：
   接收模块解析完串行数据后，输出一个并行的 8-bit 数据（`rx_data`）。把这 8 根线直接连到发送模块的输入数据端口（`tx_data`）。**实现数据的无缝转移。**

2. **`rx_done` 接 `tx_en`**：
   仅仅传数据是不够的，发送模块需要知道“什么时候开始发”。接收模块在成功收到 1 个字节的最后一个周期，会产生一个高电平脉冲（`rx_done`）。把这个脉冲直接连到发送模块的使能端（`tx_en`）。**实现收发时序的完美同步。**

> 📝 **核心结论：** 
> RTL 设计不仅仅是写逻辑，更是在“连线”。规范的命名和正确的例化，能确保硬件信号在各个模块间安全、准确地流转。
### 10.4.7 踩坑与排查：连续收发时数据丢失/乱码的修复过程
在完成了基础的串口回环代码后，实际上板测试时我遇到了一个非常经典的初学者问题：
**单次发送一个字符（如 `A`）时，收发完全正常；但当在串口助手中连续发送一长串字符（如 `Hello World`）或发送文件时，FPGA 发回来的数据会出现严重的“丢包”或“乱码”（例如变成了 `HloWrd`）。**
经过对 RTL 代码和波形进行深入排查，我定位到了问题所在，并对代码进行了针对性修改。以下是排查过程与最终的修复方案。
---
#### 📌 原因分析：为什么会丢数据/乱码？
1. **外部信号未同步（引发亚稳态与误触发）**
   `uart_rx` 信号来自 FPGA 外部引脚，属于**异步信号**。如果没有经过同步处理直接接到内部状态机中，一旦出现毛刺（Glitch），极易被错误识别为“起始位”，导致采样错位，从而产生**乱码**。
   
2. **下降沿检测不够稳定**
   最开始我仅用了一级寄存器去判断 `uart_rx` 的下降沿，但这在高速时钟下非常不可靠。
3. **连续收发的时序冲突（导致丢包的最根本原因）**
   在回环测试中，RX 接收完 1 个字节产生 `rx_done`，直接触发 TX 的 `tx_en`。
   - RX 模块通常在**停止位的正中间**（采样点）就会判定接收完成并拉高 `rx_done`。
   - 但是此时，TX 模块可能**还在发送上一个字节的停止位**！
   - 如果 TX 模块的状态机设计为“必须处于绝对的 IDLE 状态才能接收下一个 `tx_en`”，那么连续发送时，TX 就会因为“正在忙（Busy）”而直接**无视**掉新到来的 `rx_done` 脉冲，导致第二个字节直接丢失。
---
#### 🛠️ 我的修改方案（代码级修复）
针对上述问题，我从**信号同步**和**收发时序握手**两个方面进行了修改：
##### 修复 1：为异步输入引脚 `uart_rx` 打两拍 + 边缘检测
在 `uart_rx` 模块中，我增加了 3 级寄存器。前两级用于消除亚稳态（同步到系统时钟域），第三级用于稳定可靠地检测下降沿（起始位）。
```verilog
// ✅ 修复后的 UART_RX 同步与下降沿检测逻辑
reg rx_d1, rx_d2, rx_d3;
always @(posedge sys_clk or negedge sys_rst_n) begin
    if (!sys_rst_n) begin
        rx_d1 <= 1'b1;
        rx_d2 <= 1'b1;
        rx_d3 <= 1'b1;
    end else begin
        rx_d1 <= uart_rx; // 第1拍：锁存外部输入
        rx_d2 <= rx_d1;   // 第2拍：消除亚稳态，同步到系统时钟域
        rx_d3 <= rx_d2;   // 第3拍：用于打拍延迟，配合寻找下降沿
    end
end
// 检测下降沿：当上一拍为1（高电平），当前拍为0（低电平）时，说明出现下降沿
wire rx_fall_edge = rx_d3 & (~rx_d2);
```
> **效果：** 彻底解决了因线路噪声或亚稳态导致的乱码问题，起始位抓取变得极其精准。

##### 修复 2：优化波特率采样点（严格取中点）
确保数据的采样不仅在对应的数据位，还要严格卡在波特率计数的**正中间（`BPS_CNT / 2`）**。这样即使电脑端和 FPGA 端的时钟存在微小的频率偏移，也能保证 8 个数据位采样不会越界。

##### 修复 3：在 TX 模块中增加对连续发送的时序容忍度
为了防止 RX 的完成脉冲到来时 TX 还在发送停止位而被丢弃，我修改了 TX 状态机中停止位（STOP）跳转到起始位（START）的逻辑：
**不再强求必须回到 IDLE 状态。如果在发送停止位即将结束时检测到新的 `tx_en`，允许状态机无缝衔接到下一个 START 状态。**

*(注：在更复杂、更稳妥的企业级工程中，解决丢包最根本的办法是在 RX 和 TX 之间加入一个 **FIFO（先入先出缓存器）**。RX 把数据写进 FIFO，TX 根据自己的节奏从 FIFO 里读数据，这样无论电脑发多快，只要 FIFO 不满，就不会丢数据。)*

---

#### 💡 修复总结

| 故障现象 | 根本原因 | 我的修复方法 |
| :--- | :--- | :--- |
| **偶发乱码** | 异步信号引入毛刺，误触发起始位 | 使用 `rx_d1`、`rx_d2` **打两拍**消除亚稳态 |
| **无法检测起始位** | 边沿检测逻辑不可靠 | 使用 `rx_d3 & (~rx_d2)` 组合逻辑**精准提取下降沿脉冲** |
| **连续发送丢失字符** | 收发时序错位，TX 处于 Busy 漏接使能脉冲 | 优化 TX 状态机停止位跳转逻辑（进阶方案可引入 **FIFO 缓存**） |

> 经历了这次排查与修改，我深刻体会到了**“仿真通过不等于上板成功”**。真实的硬件世界存在异步、毛刺和时钟偏移，RTL 代码必须具备足够的鲁棒性（Robustness）才能应对这些物理特性。

```verilog
    input  wire sys_clk,
    input  wire sys_rst_n,
    input  wire uart_rx,
    output wire uart_tx
);
wire [7:0] rx_data;

reg        tx_start;
wire       tx_busy;
reg  [7:0] data_buf;
reg        data_valid;
// 接收到数据后打入缓存，等待发送器空闲后发送（串口回环逻辑）
always @(posedge sys_clk or negedge sys_rst_n) begin
    if(!sys_rst_n) begin
        tx_data    <= 8'd0;
        tx_start   <= 1'b0;
        data_buf   <= 8'd0;
        data_valid <= 1'b0;
    end
    else begin
        tx_start <= 1'b0;
        // 收到新数据，存入缓存
        if(rx_done) begin
            data_buf   <= rx_data;
            data_valid <= 1'b1;
        end
        // 发送器空闲，且有有效数据时触发发送
        if(data_valid && !tx_busy) begin
            tx_data    <= data_buf;
            tx_start   <= 1'b1;
            data_valid <= 1'b0; // 清除有效标志
        end
    end
end

    .CLK_FREQ(50000000),
    .BAUD    (115200)
) u_uart_rx (
    .sys_clk  (sys_clk),
    .sys_rst_n(sys_rst_n),
    .uart_rx  (uart_rx),
    .rx_data  (rx_data),
    .rx_done  (rx_done)
);
// 例化发送模块
uart_tx #(
```

```verilog
`timescale 1ns / 1ps

module uart_rx #(
    parameter CLK_FREQ = 50000000,
    parameter BAUD     = 115200
)(
    input  wire       sys_clk,
    input  wire       sys_rst_n,
    input  wire       uart_rx,
    output reg [7:0]  rx_data,
    output reg        rx_done
);
localparam BAUD_CNT_MAX  = CLK_FREQ / BAUD;
localparam BAUD_CNT_HALF = BAUD_CNT_MAX / 2;
reg uart_rx_d0;
reg uart_rx_d1;
reg rx_busy;
reg [15:0] baud_cnt;
reg [3:0]  bit_cnt;
reg [7:0]  rx_data_t;
wire rx_fall;
// 下降沿检测
assign rx_fall = (uart_rx_d1 == 1'b1) && (uart_rx_d0 == 1'b0);
always @(posedge sys_clk or negedge sys_rst_n) begin
    if(!sys_rst_n) begin
        uart_rx_d0 <= 1'b1;
        uart_rx_d1 <= 1'b1;
    end
    else begin
        uart_rx_d0 <= uart_rx;
        uart_rx_d1 <= uart_rx_d0;
    end
end

always @(posedge sys_clk or negedge sys_rst_n) begin
    if(!sys_rst_n) begin
        rx_busy   <= 1'b0;
        baud_cnt  <= 16'd0;
        bit_cnt   <= 4'd0;
        rx_data_t <= 8'd0;
        rx_data   <= 8'd0;
        rx_done   <= 1'b0;
    end
    else begin
        rx_done <= 1'b0;
        if(!rx_busy) begin
            baud_cnt <= 16'd0;
            bit_cnt  <= 4'd0;
            if(rx_fall) begin
                rx_busy  <= 1'b1;
            end
        end
        else begin
            baud_cnt <= baud_cnt + 1'b1;
            if(bit_cnt == 4'd0) begin
                if(baud_cnt == BAUD_CNT_HALF - 1'b1) begin
                    baud_cnt <= 16'd0;
                    if(uart_rx_d1 == 1'b0) begin
                        bit_cnt <= 4'd1;
                    end
                    else begin
                        rx_busy <= 1'b0;
                    end
                end
            end
            else if((bit_cnt >= 4'd1) && (bit_cnt <= 4'd8)) begin
                if(baud_cnt == BAUD_CNT_MAX - 1'b1) begin
                    baud_cnt <= 16'd0;
                    case(bit_cnt)
                        4'd1: rx_data_t[0] <= uart_rx_d1;
                        4'd2: rx_data_t[1] <= uart_rx_d1;
                        4'd3: rx_data_t[2] <= uart_rx_d1;
                        4'd4: rx_data_t[3] <= uart_rx_d1;
                        4'd5: rx_data_t[4] <= uart_rx_d1;
                        4'd6: rx_data_t[5] <= uart_rx_d1;
                        4'd7: rx_data_t[6] <= uart_rx_d1;
                        4'd8: rx_data_t[7] <= uart_rx_d1;
                        default: rx_data_t <= rx_data_t;
                    endcase
                    bit_cnt <= bit_cnt + 1'b1;
                end
            end
            else if(bit_cnt == 4'd9) begin
                if(baud_cnt == BAUD_CNT_MAX - 1'b1) begin
                    baud_cnt <= 16'd0;
                    rx_busy  <= 1'b0;
                    bit_cnt  <= 4'd0;
                    if(uart_rx_d1 == 1'b1) begin
                        rx_data <= rx_data_t;
                        rx_done <= 1'b1;
                    end
                end
            end
        end
    end
end
endmodule
```
